# Agent Sandbox Landscape: What the Scores Actually Tell Us

*2026-03-08 · 23 products scored across 7 defense layers*

---

## Nobody Governs the Agent

The single most striking pattern in the data: **70% of products have no runtime action governance** (L6 ≤ 1 or null). Thirteen products don't address it at all. One (packnplay) explicitly scores zero — the container *is* the permission boundary, and agents run with `--dangerously-skip-permissions`. The flag is right there in the name.

The reason is a taxonomy distinction that matters: HITL approval, PR checkpoints, and seccomp profiles all *feel* like governance but aren't scored as L6. Seccomp restricts syscalls (that's compute isolation, L1). A human reviewing a PR is the human providing governance, not the sandbox. When you strip those away, only **7 products** have the sandbox itself enforcing what the agent can do with its access — and only one (nono) reaches kernel-level enforcement.

This means most sandboxes will happily let an agent `kubectl delete namespace production` as long as it has network access and credentials. The box is strong. The agent inside it is unsupervised.

## The Inverse Correlation Nobody Wanted

Products that invest the most in compute isolation invest the least in everything else. Of the 10 products with L1:4 (microVM/hardware isolation), **7 have an upper-layer average below 2.0**. E2B, Fly.io Sprites, Blaxel, and Unikraft Cloud all hit L1:4 / L2:4 / L3:4 — and then L5:1 and L6:null. Devin manages L1:4 with L4:0 — a Firecracker microVM with unrestricted internet access and secrets as env vars.

The products with the strongest upper layers (nono, Google Agent Sandbox, Stakpak Warden) all have L1 ≤ 3. It's as if the industry split into two camps: "build a better box" and "control what's inside the box" — and nobody's in both.

The three exceptions that balance both: **Copilot coding agent** (4/4/4/2/2/2/2), **Deno Sandbox** (4/4/3/3/3/-/2), and **Claude Code web** (4/-/4/2/4/-/2). Though Claude Code web's L6 is null — the agent is autonomous within its session.

## Credentials: Half the Market Hands Over the Keys

48% of products (11/23) expose raw credentials inside the sandbox at S:0–1. The agent gets your API keys as env vars and can do whatever it wants with them. Combined with the 5 products that have L4 ≤ 1 (weak or no network boundary), the exfiltration math is simple: read secret + send HTTP request = game over.

The credential management spectrum has a visible gap. Two products reach S:4 (Docker Sandbox and Claude Code web — credentials literally never enter the sandbox). Four reach S:3 (Google, Warden, nono, Deno Sandbox — placeholder substitution or proxy injection). Then there's a cliff: six at S:2, and eleven at S:1 or below. The middle tiers are underpopulated. Most products either solved this properly or didn't try.

Devin is the most interesting case: it has a sophisticated Secrets Manager with org/repo/session scopes, TOTP support, and RBAC — but secrets are still accessible as normal env vars inside the Devbox. Granularity G:3, strength S:1. Fancy key management, cooperative enforcement.

## T3-Remote: The Threat Almost Nobody Addresses

Preventing an agent from destroying *remote* infrastructure (T3-Remote) requires both network control (L4 ≥ 2) AND action governance (L6 ≥ 2). Only **7 out of 23 products** achieve this. Five products score T3-Remote as completely undefended — not partial, *none*.

This is the sharpest differentiator in the dataset. Every product with L1 ≥ 2 and L3 ≥ 2 handles local destructive operations (T3-Local) — that's table stakes. But remote destruction is a network operation, and network filtering alone doesn't help if the agent has legitimate access to AWS and decides to call `DeleteStack`. You need semantic governance: "you can reach AWS, but you cannot delete things." Almost nobody enforces that.

## nono Is the Universal Complement (In Theory)

nono's fingerprint is `3/-/3/3/3/3/3` — S:3 across every layer it addresses, missing only L2 (resource limits). When you compose it (take the max at each layer) with any product that has L2 ≥ 2, you get 7/7 threat coverage. That's **15 out of 22 other products**. No other product comes close to this composability.

The catch: it's early alpha, local-only, and you'd need to run it inside or alongside the sandbox. But the fingerprint math is hard to argue with. It fills exactly the layers that cloud platforms leave open (L4–L7), while cloud platforms fill its only gap (L2).

The best theoretical local stack: **Docker Sandbox + nono = 4/4/4/3/4/3/3**. The best cloud stack: **E2B + nono = 4/4/4/3/3/3/3**.

## Granularity Without Strength Is Security Theater

Fifteen instances across the dataset where a product's granularity score exceeds its strength score at a given layer. Devin at L5: S:1, G:3 — you can configure secrets with org/repo/session scopes and TOTP and JSON and cookies, but the agent sees them as plain env vars. StrongDM Leash at L4: S:2, G:3 — Cedar policies can express per-connection context-aware rules, but enforcement is capped at S:2 because L1 is a shared-kernel container.

The pattern: products build sophisticated policy UIs and configuration surfaces, then enforce them through mechanisms the agent could theoretically circumvent. The controls are real, but the enforcement ceiling is lower than the granularity suggests.

## The Local-First Desert

Seven products run locally. Five of them have L2 = 0 or null — no resource limits at all. A fork bomb in Claude Code local, Codex CLI, nono, Cursor, or StrongDM Leash will take down your machine. The only local products with resource limits are Docker Sandbox (hypervisor-enforced) and... that's it for S ≥ 2.

Cloud-managed products dominate: 14/23. If you want full-stack sandbox coverage without sending your code to someone else's infrastructure, your options are thin and require composition.

## Every Score Is Unverified

Every single product has `evidence_level: docs`. One product (Ona) has a `last_reviewed` date. The rest are `null`. All scores are based on documentation and marketing materials — not hands-on testing, not source code review, not bypass attempts. The actual security posture of any product could be worse than reported. Blaxel's entry explicitly flags a "blog-to-docs gap on security features." Ona has four separate review flags across its layers.

The repo includes `ast-probe`, a binary you can drop into any sandbox to get verified scores. Nobody has used it yet.
